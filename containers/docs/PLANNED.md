# Planned Container Tools

Purpose: track planned VCF-downstream tool onboarding before promotion to `experimental` or `production`.

[../README.md](../README.md), [../index.md](../index.md), and
[TOOL_LIFECYCLE.md](TOOL_LIFECYCLE.md) define the adjacent container control
surfaces for planned tools.

Scope: tools with `status = "planned"` in
[configs/ci/registry/tool_registry_vcf_downstream.toml](../../configs/ci/registry/tool_registry_vcf_downstream.toml).

Non-goals:
- Declaring production support.
- Replacing registry contracts.

Contracts:
- Planned tools must appear in
  [configs/ci/tools/images.toml](../../configs/ci/tools/images.toml) with
  `enabled = false`.
- Planned tools must appear in
  [containers/versions/versions.toml](../versions/versions.toml) with
  `status = "planned"`.
- Every registry-planned tool must appear exactly once in the planned tool table below.
- Missing container defs are allowed for planned tools and forbidden for production tools.

## VCF Downstream Stack Admission Checklist

For each tool below, promotion from `planned` to `experimental`/`production` requires:
- pinned upstream version and checksum/commit in
  [containers/versions/versions.toml](../versions/versions.toml)
- apptainer + docker arm64 definitions (or explicit runtime exemption)
- smoke coverage for `--version`, `--help`, and minimal command contract
- registry metadata wired in
  [configs/ci/registry/tool_registry_vcf_downstream.toml](../../configs/ci/registry/tool_registry_vcf_downstream.toml)
- QA matrix row generated by `cargo run -p bijux-dna-dev -- containers run generate-qa-matrix`
