# HPC Lunarc Layout

This project uses a fixed four-directory Lunarc layout rooted under `/home/bijan/bijux`.

## Directories

- `/home/bijan/bijux/bijux-dna`
- `/home/bijan/bijux/bijux-dna-container`
- `/home/bijan/bijux/corpus_01`
- `/home/bijan/bijux/results`

## Invariants

1. `bijux-dna` is code only (git workspace, configs, scripts).
2. `bijux-dna-container` stores container assets (SIF files, cache state).
3. `corpus_01` stores the governed benchmark corpus and its normalized FASTQ material.
4. `results` stores all run outputs and run-scoped temporary data.
5. HPC profile config is generated by `bijux config init-hpc` into `configs/runtime/profiles/hpc.toml` and is not hand-edited.
6. Every HPC run must carry reproducibility metadata and run-context metadata in `run_manifest.json`.
7. Site environment lock metadata is written to `results/site_lock.json`.
8. Paths under `/home/bijan/bijux/results` follow: `<corpus>/<pipeline>/<stage>/<tool>/<timestamp>/<run_id>`.
9. No shared temp directories: each run uses a unique temp path under results or scratch, keyed by run id.
10. The summed footprint of `bijux-dna`, `bijux-dna-container`, `corpus_01`, and `results` on the Lunarc frontend must stay below 20 GB.
11. Pull Lunarc results into the local evidence mirror rooted at `/Users/bijan/bijux/bijux-dna-results`.

## Commands

- Initialize HPC profile/layout:
  - `bijux config init-hpc --root /home/bijan/bijux`
- Validate HPC status:
  - `bijux status --hpc`
- Export HPC SIF inventory and digests:
  - `bijux environment export-hpc --json`
- Pull Lunarc results into the local mirror:
  - `make pull-lunarc-results`
- Pull results and clear the remote results payload:
  - `make pull-lunarc-results-prune`
- Audit the frontend footprint:
  - `make lunarc-footprint`
- Clear transient build residue from the frontend repo checkout:
  - `make lunarc-prune-code`

## What
Defines the fixed HPC directory layout and invariants for deterministic `bijux-dna` execution at Lunarc.

## Why
Separating code, data, containers, and results avoids cross-contamination and improves reproducibility.

## Non-goals
- Describe non-Lunarc cluster layouts.
- Replace per-pipeline runbooks.

## Contracts
- Runs use only the declared four roots.
- Reproducibility metadata is written for each run.
- Result paths remain run-scoped and timestamped.

## Examples
- `bijux config init-hpc --root /home/bijan/bijux`
- `bijux status --hpc`

## Failure modes
- Missing one of the required roots causes profile validation failures.
- Shared temp/output paths can break reproducibility guarantees.
